# MCaml datapack tooling

This directory holds the host-side scripts that wrap MCaml's compiler
output into a loadable Minecraft datapack. See
`~/.claude/plans/mcaml-datapack-packaging.md` for the design.

## Build a datapack

Two-step flow (Stage 5/6 of the plan; the OCaml side has no
`--datapack` flag in v1):

```sh
# 1. compile to a flat directory of .mcfunction files
./mcaml -o build/ < scripts/tests/test_full_chain.mcaml

# 2. wrap the directory into a real datapack
python3 tools/pack_datapack.py \
    --input build/ \
    --name mcaml_test \
    --output dist/mcaml_test/        # or dist/mcaml_test.zip
```

The packager copies every `*.mcfunction` from `--input` into
`data/mcaml/function/` and synthesizes:

- `pack.mcmeta` — `pack_format: 41` (1.20.5 baseline) with
  `supported_formats: [41, 81]` so the same pack loads on 1.21.x.
- `data/mcaml/function/init.mcfunction` — registers the `vars`
  scoreboard objective, zeroes the `mcaml:stk frames` and
  `mcaml:tmp args` storage roots, and sets
  `gamerule maxCommandChainLength 10000000` (override with
  `--chain-length N`; `--chain-length 0` leaves the game rule alone).
  Raising the rule is load-bearing, not belt-and-braces: vanilla counts
  **every** command in a function chain against the limit, including
  `execute if/unless` lines whose conditions fail, and MCaml's
  guard-chain codegen means a compiled if/match dispatch always runs
  its full command count regardless of which arm is taken. One
  scheduled step of a nontrivial program can therefore exceed the
  65536 default and silently stop mid-chain — the graph-viz demo's
  bfs/dijkstra froze exactly this way on 1.21.4 (`sim.py` counts only
  taken commands and never sees it).
- `data/minecraft/tags/function/load.json` — wires `mcaml:init` to
  `#minecraft:load` so the runtime is set up at world load.

The compiler refuses any program that defines a top-level function
named `init` so user code can't clobber the synthesized loader.

## Entrypoint convention (chat-driven invocation)

Once the datapack is dropped into a save's `datapacks/` directory and
`/reload`'d, every compiled top-level function is callable from chat as
`mcaml:<function_name>`. The runtime uses fixed scoreboard slots for
parameters and the return value:

| slot      | objective | role                                |
|-----------|-----------|-------------------------------------|
| `param_0` | `vars`    | first scalar argument               |
| `param_1` | `vars`    | second scalar argument              |
| `param_N` | `vars`    | Nth scalar argument                 |
| `$ret`    | `vars`    | scalar return value                 |

To call `fun add(a: int, b: int): int = a + b` with `a=7`, `b=5`:

```
/scoreboard players set param_0 vars 7
/scoreboard players set param_1 vars 5
/function mcaml:add
/scoreboard players get $ret vars
```

The last command prints the return value to chat.

`unit`-returning functions still write `$ret` (the codegen treats unit
as `0`); you can ignore it. `cmd!` strings — including `say` — execute
in-line, so output lands in chat as the function runs.

### Worked example: `test_full_chain.mcaml`

This program is parameterless: `main_test()` builds two length-4 arrays
inline and calls `dot`. The chat sequence is just:

```
/function mcaml:main_test
/scoreboard players get $ret vars
```

Expected: `$ret = 70` (`1*5 + 2*6 + 3*7 + 4*8`). Cross-check against
`/tmp/mcaml_out/sim.py` for the same source — both runtimes must
agree.

### Array-taking entrypoints

Currently impossible. Arrays are not first-class runtime values in
M1/M2: a `let a = [| ... |]` binds `a` to a compile-time storage ID,
and arrays can't cross function boundaries from the chat side. Once
microGPT lands and we need real array I/O, we'll add a per-entrypoint
wrapper file that writes the input to `storage mcaml:heap <aid>`,
calls the function, and reads back from the result aid. Out of scope
for v1.

## Graph algorithm visualization demo

An animated in-world demo of six graph algorithms (BFS, DFS, Dijkstra,
Prim, Kruskal, Edmonds-Karp max-flow/min-cut). Two graphs are built out
of wool node pads, concrete edge lines, and `text_display` weight
labels; each algorithm recolors blocks one logical step per second
so you can watch it work. Assumes any flat area (the spawn function
fills its own platform at Y=57, the ground-surface level of the
Redstone Ready superflat preset, whose player spawn is (0, 58, 0)).

Sources: `scripts/demos/graph_world.mcaml` is GENERATED by
`tools/gen_graph_viz.py` (layouts, palette, and coordinates all live in
the generator — rerun it after any layout change); the algorithms are
hand-written in `scripts/demos/graph_algos.mcaml`. Offline verification:
`tools/sim_check_graph.py` (also wired into `verify_canary.py`).

Build and pack:

```sh
python3 tools/gen_graph_viz.py        # only after editing the generator
cat scripts/demos/graph_world.mcaml scripts/demos/graph_algos.mcaml | ./mcaml -o build_graph
python3 tools/sim_check_graph.py build_graph
python3 tools/pack_datapack.py --input build_graph --name mcaml_graph \
    --output dist/mcaml_graph.zip
```

In-game (drop the zip into the save's `datapacks/`, then):

```
/reload
/function mcaml:graph_spawn       # build both graphs at spawn (y=58 surface)
/function mcaml:bfs_start         # frontier wave from node 0
/function mcaml:dfs_start         # probing walk with backtracking
/function mcaml:dijkstra_start    # settles nodes, lime tree at the end
/function mcaml:prim_start        # MST grown from node 0 (weight 135)
/function mcaml:kruskal_start     # global cheapest-edge MST, red = rejected
/function mcaml:ek_start          # max flow S->T; ends painting the min cut
/function mcaml:graph_despawn     # remove everything
```

Each `<algo>_start` resets the colors and state itself, so runs can be
repeated or interleaved freely; starting a new algorithm cancels a
running one (its next scheduled step sees a foreign token in `st[15]`
and does nothing). Progress/state readable from chat:
`$graph_done` (1 when finished) and, after `ek_start`, `$graph_flow`
(the max-flow value, expected 40) via
`/scoreboard players get <slot> vars`.

Colors: white = idle, yellow = frontier/path, orange = current /
considering, lime = visited/accepted, red = rejected/min-cut,
purple = saturated flow edge, light blue / pink = min-cut sides,
blue/red pads = flow source/sink. The step delay is the literal
`20t`/`40t` in the `schedule` strings in `scripts/demos/graph_algos.mcaml`.
Graph size and layout are generator-only changes (`tools/gen_graph_viz.py`
emits the `gsz` size table the algorithms read their loop bounds from).

Note: the sim validates the algorithms and the recolor command stream
(`World.viz`), but Minecraft itself is the only parser of the command
strings — after changing block ids or `text_display` NBT in the
generator, do one manual `/reload` pass per `tools/MANUAL_TEST.md`.
